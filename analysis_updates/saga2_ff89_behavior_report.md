# SaGa2 FF89 behavior report

## 対象

- `rom/SaGa 2 - Hihou Densetsu (J) (V1.1).gb`
- 既存 `saga2_rng_wait_io_report.md`

## 目的

- `FF89` が何を表すかを、参照箇所の局所文脈から絞る
- `00:0469` 周辺の意味を一段具体化する

## 結論

`FF89` は単純な RNG 状態ではなく、**入力ビット列または入力由来の状態値** と見るのが自然。
特に

- `00:1761` で複数値を合成して `FF89` に書く
- `00:1766` で `FF89 & 0x0F` を判定する
- `00:29A3` で `FF89` から 0..4 の小さい方向 index 風値を作る
- `01:68F5` で `BIT 0, FF89` を直接見る

という使われ方があり、**ボタン / 方向入力の正規化済み状態** 候補が最も強い。

## 1. `00:0469` 周辺

```text
0469: LDH A,($FF89)
046B: LD C,A
046C: LD A,($C775)
046F: CP C
0470: JR NZ,$16
0472: LD A,($C774)
0475: DEC A
0476: JR Z,$07
0478: LD ($C774),A
047B: XOR A
047C: LDH ($FF8A),A
047E: RET
0480: LD A,$05
0482: LD ($C774),A
0485: LDH ($FF8A),A
0487: RET
0488: LD A,$1E
048A: LD ($C774),A
048D: LDH ($FF8A),A
048F: LD ($C775),A
0492: RET
```

ここでは `FF89` を `C775` と比較し、同一値が続くかどうかで

- `C774` に `1E` または `05` の repeat 間隔を設定
- `FF8A` には `0` または `FF89` を入れる

構造になっている。これは乱数値よりも、**現在入力と前回入力の比較** に近い。

## 2. `00:1761` / `00:1766`

```text
175C: ...
175F: AND (HL)
1760: AND (HL)
1761: LDH ($FF89),A
1763: LD A,$30
1765: LDH (C),A
1766: LDH A,($FF89)
1768: AND $0F
176A: CP $0F
176C: JP Z,$0200
```

観察点:

- `FF89` はここで RAM/IO 値の合成結果として書かれる
- 直後に low nibble (`& 0x0F`) を取り、`0x0F` と比較している

このパターンは「下位 4bit に方向キー群が入り、全部 1 なら無入力」のような
**入力マトリクス / joypad nibble** 的解釈と整合しやすい。

## 3. `00:29A3`

```text
29A3: LDH A,($FF89)
29A5: LD B,$04
29A7: RLCA
29A8: JR C,$05
29AA: DEC B
29AB: JR NZ,$FA
29AD: LD B,$05
29AF: LD A,$05
29B1: SUB B
29B2: RET
```

このルーチンは `FF89` を回転しながら最初に立つ bit を探し、
最終的に `A=0..4` の小さい値を返す。

自然な読み:

- 4方向のどれか 1 つを direction index へ変換
- 該当なしなら特別値 `4`

したがって `FF89` には、**少なくとも方向情報として扱える bit 配置** が含まれている。

## 4. `01:68B5`

```text
68B5: LDH A,($FF89)
68B7: AND $0C
68B9: CP $0C
68BB: JP Z,$699F
```

ここでは `FF89` の bit2/bit3 相当だけを抜いて判定している。
これも「複数の個別入力 bit を見る」使い方であり、乱数状態値には見えにくい。

## 5. `01:68F5`

```text
68F5: LDH A,($FF89)
68F7: BIT 0,A
68F9: JR Z,$04
68FB: LD A,$01
68FD: LDH ($FF90),A
```

bit 単位で直接見るので、`FF89` は

- packed button state
- packed direction state
- あるいはその正規化結果

のいずれかと見るのが妥当。

## 6. 現時点の整理

### 確度が高いこと

- `FF89` は RNG seed/state ではない
- `FF89` は bitfield 的に扱われている
- `00:0469` は `FF89` を前回値 `C775` と比較し、入力継続/変化に応じた結果コードを作る可能性が高い
- `00:29A3` は `FF89` から方向 index 風の値を作る変換ルーチン候補

### まだ未確定なこと

- active-high / active-low の向き
- bit0-3 と bit4-7 の正確な割当
- `FF8A = 00/05/1E` が具体的に何フレーム / 何状態を意味するか

## 次の一手

1. `00:0469` を擬似コード化し、`C774/C775` を含めて状態機械として書く
2. `FF89` 書き込み元をさらに追って、joypad 生値との対応を取る
3. `FF8A` を使う script/battle 側の待機条件と照合する
